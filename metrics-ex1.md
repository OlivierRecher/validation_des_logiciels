## Do you feel its size roughly matches its responsibility?

### Bank

No, this class violates the Single Responsibility Principle. While it correctly manages the list of accounts, it also improperly handles data persistence in the saveAccounts method (lines 141–172). Hardcoding file paths and writing to disk is a separate responsibility that belongs in a storage or repository service, not the account manager.

### BankAccount

No, the class contains logic that does not belong to a simple bank account entity. Specifically, the loadFromText method (lines 157–204) includes file reading and parsing logic. A domain object should store state (balance, owner) and behavior (deposit, withdraw), but it should not be responsible for reading files from the disk.

### Person

No, the class contains logic that does not belong to a simple person entity. Specifically, the constructor (lines 92–110) includes parsing logic for a raw string into a Person object. A domain object should theoretically just hold data (Name, Age, etc.). The logic to parse a raw string string into a Person object should reside in a utility method or a factory, not inside the entity itself.

### BankAccountApp

No. While closer to being correct than the others, it still takes on too much responsibility by including parsing logic in its `main()` method (lines 92–110). The application should orchestrate workflow without embedding complex parsing logic. The logic to parse raw strings into `Person` and `BankAccount` objects should reside in utility methods or factories, not inside the entry point itself.
